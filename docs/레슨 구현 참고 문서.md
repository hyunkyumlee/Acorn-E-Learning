## 1. 재검증 결론

### 확정 흐름

```text
과목 Subject
└─ 난이도/레벨 Bronze, Silver, Gold
   ├─ 행성/단원 1
   │  ├─ 레슨 1
   │  │  ├─ 이론 학습
   │  │  └─ 일반 문제 10개
   │  ├─ 레슨 2
   │  ├─ ...
   │  └─ 레슨 10
   ├─ 행성/단원 2
   ├─ 행성/단원 3
   ├─ 행성/단원 4
   ├─ 행성/단원 5
   └─ Gate: AI 코딩테스트 3문제
      └─ 2문제 이상 통과하면 다음 난이도 해금
```

### 용어 정정

`세트 생성`은 문제 원본을 새로 만드는 뜻이 아니다.

```text
문제 원본 생성/등록: practice_problems에 문제를 만드는 작업
세트 생성: 사용자가 "이번 레슨 문제풀이 1회차"를 시작하는 작업
문제 순서 저장: 이번 풀이 회차에서 1~10번에 어떤 문제를 보여줄지 practice_set_items에 저장하는 작업
```

즉 고지연님 범위는 문제 생성이 아니라 아래 흐름이다.

```text
lessonId=101 문제풀이 시작
→ practice_problems에서 lesson_id=101 문제 10개 조회
→ practice_set_attempts에 풀이 회차 생성
→ practice_set_items에 10문제 순서 저장
→ 사용자가 1~10번 문제 풀이
→ practice_submissions에 답안 저장
→ 7개 이상 정답이면 user_lesson_progress.practice_passed=1
```

## 2. DB 기준

이미 DDL/샘플데이터에는 아래 기준이 반영되어 있다.

| 테이블 | 역할 |
|---|---|
| `lessons` | 행성 하위 레슨. `required_for_completion=1`인 레슨만 행성 완료 계산에 포함 |
| `user_lesson_progress` | 사용자별 레슨 이론/문제풀이 완료 상태 |
| `practice_problems` | 문제 원본. 레슨별 문제는 `lesson_id`로 묶음 |
| `practice_set_attempts` | 사용자의 문제풀이 1회차 |
| `practice_set_items` | 해당 회차에 포함된 문제 10개와 순서 |
| `practice_submissions` | 사용자가 제출한 답안. 가능하면 `set_item_id`까지 저장 |
| `learning_progress` | 기존 로드맵 화면 호환용 행성 단위 aggregate로 유지 가능 |

권장 source of truth:

```text
레슨 완료 여부: user_lesson_progress
행성 완료 여부: 해당 행성의 required lesson 전체가 theory_completed=1 AND practice_passed=1인지 집계
로드맵 표시: user_lesson_progress 집계 결과를 learning_progress에 반영하거나, 집계 쿼리로 직접 계산
```

### 구현 방식

1. 레슨 선택 화면 진입

```text
GET /learning/nodes/{nodeId}/lessons
```

화면에 필요한 데이터:

```text
node
lessons
userLessonProgressMap
completedRequiredLessonCount
totalRequiredLessonCount
```

2. 문제풀이 시작

```text
POST /learning/practice/sets
body/form: lessonId
```

서버 처리:

```text
lessonId로 lessons 조회
lesson.nodeId로 curriculum_nodes 조회
practice_problems where lesson_id = :lessonId and is_active=1 order by problem_id limit 10
practice_set_attempts insert(userId, subjectId, nodeId, lessonId, totalCount=10)
practice_set_items insert 10 rows(setAttemptId, problemId, sortOrder)
첫 문제 화면으로 redirect
```

3. 답안 제출/완료

```text
각 문제 제출은 session 또는 practice_set_items 순서 기준으로 진행
마지막 문제 후 complete
정답 7개 이상이면:
  user_lesson_progress.practice_passed=1
  progress_rate=100 if theory_completed=1 else 50
  출석/점수 처리
```

4. 완료 후 버튼 계산

```text
현재 lesson의 nodeId 안에서 sort_order가 더 큰 다음 required lesson 조회
다음 레슨이 있고 행성 전체 완료가 아니면:
  다음 레슨 버튼 노출 → /learning/lessons/{nextLessonId}
항상:
  학습 메인으로 버튼 노출 → /learning
행성 전체 required lesson 완료면:
  다음 레슨 버튼 숨김
  자동/버튼 이동 모두 /learning 기준
```

### 구현 방식

로드맵 상태 계산은 아래 기준으로 맞춘다.

```text
행성 완료:
  count(required lessons) == count(required lessons where theory_completed=1 and practice_passed=1)

행성 진행률:
  완료 required lesson 수 / 전체 required lesson 수

Gate ready:
  선택 subject + level의 PLANET 5개가 모두 완료
```

주의:

```text
기존 learning_progress는 다른 화면에서 쓰고 있으므로 바로 제거하지 않는다.
단, 새 구조의 source of truth는 user_lesson_progress로 둔다.
필요하면 user_lesson_progress 집계 후 learning_progress를 aggregate mirror로 upsert한다.
```

### 구현 방식

Gate 응시 가능 쿼리 기준:

```sql
-- 개념 예시
SELECT COUNT(*) AS incomplete_required_lesson_count
FROM lessons l
JOIN curriculum_nodes cn ON cn.node_id = l.node_id
LEFT JOIN user_lesson_progress ulp
  ON ulp.lesson_id = l.lesson_id
 AND ulp.user_id = :userId
WHERE cn.subject_id = :subjectId
  AND cn.level_code = :levelCode
  AND cn.node_type = 'PLANET'
  AND cn.is_active = 1
  AND l.is_active = 1
  AND l.required_for_completion = 1
  AND NOT (
    COALESCE(ulp.theory_completed, 0) = 1
    AND COALESCE(ulp.practice_passed, 0) = 1
  );
```

`0`이면 Gate 응시 가능.
